Basic Project Questions

QWhat is PairOn?

PairOn is a real-time collaborative developer platform built for students and developers who want to code together, learn from each other, and build projects — all directly in the browser.

It combines a smart matchmaking system (like Tinder for coders), a full browser-based IDE (Monaco Editor + WebContainers), live pair programming, a Quick Chat system for spontaneous conversations, a Friends & DM system (like Instagram DMs), and an AI assistant — all in one unified platform. No downloads, no local setup required.

QWhat inspired you to build this project?

The inspiration came from the real frustration of solo learning. When you're stuck on a bug at 2 AM, there's no one to pair with. Platforms like LeetCode or HackerRank let you code alone, while Discord or Zoom require awkward screen sharing without a shared editor.

PairOn was inspired by the idea: "What if you could instantly find another developer, jump into a shared code environment, and build something together — as easily as starting a video call?"

The concept of pair programming has been proven to produce better code and faster learning. PairOn makes it accessible to everyone.

It's a classic 3-tier architecture (Client → Application Server → Database), enhanced with a real-time bidirectional WebSocket layer for all collaboration features.

QHow does the frontend communicate with the backend?

Two communication channels run simultaneously:

  • REST API (HTTPS) — Standard CRUD operations: login, register, fetch profile, send friend request, load message history. Uses fetch with JWT Bearer token in the Authorization header.
  • Socket.IO (WebSocket) — All real-time features: matchmaking events, collaboration chat, code sync, terminal output, typing indicators, file changes, IDE state push/pull. The socket authenticates using the same JWT passed in the handshake's auth field.
QHow does the code editor work in your platform?

The editor is Monaco Editor — the exact same engine that powers VS Code. Key implementation details:

  • Each file is stored as a Monaco ITextModel with a URI (file:///filename.tsx)
  • Switching tabs = editor.setModel(model) — no re-render, preserving cursor position and undo/redo history
  • TypeScript language server configured with jsx: react-jsx for full .tsx IntelliSense
  • React type stubs and react/jsx-runtime stubs injected as extra libs — eliminates false red underlines
  • File changes are debounced (1 second) and synced to the partner via code:file-change socket events
  • A file locking system prevents simultaneous edits on the same file — a lock icon appears if your partner is editing it
QHow does the system execute code?

Code execution happens through WebContainers — a WASM-based Node.js runtime that runs inside the browser tab itself.

  1. User's files are mounted into the WebContainer virtual filesystem
  2. npm install is run inside the WebContainer to install dependencies
  3. npm run dev starts a Vite dev server inside the browser
  4. The WebContainer exposes a local URL accessible via an <iframe> preview panel
  5. This preview URL is broadcast to the partner via socket so both can see the running app

Importantly, no server-side code execution happens — the browser is the server. This means zero server CPU cost for code running and complete isolation between users.

QHow does your AI integration work?
  1. User opens the AI panel and asks a question (e.g., "review my code", "fix this bug")
  2. The frontend sends the current file content + user's query to the backend /api/ai/assist
  3. Backend calls the Groq API (LLaMA 3 70B) with a system prompt framing the AI as a senior developer
  4. Response is streamed back and displayed in the AI panel
  5. The AI can suggest code edits, which the user can apply with one click — inserting directly into Monaco editor

Development Questions

QWhat challenges did you face while building the project?
  • WebContainer cold start — Takes ~30 seconds on first boot. Solved with a progress bar and loading feedback.
  • Real-time sync race conditions — File changes arriving out of order. Solved with sequence IDs and last-write-wins for non-conflicting edits.
  • Monaco model management — Multiple open files each need their own model. Moving between tabs without re-rendering was complex.
  • Terminal sync — Xterm.js terminals are not serializable state; syncing required relaying raw output strings character by character via sockets.
  • Socket room isolation — Ensuring two collaborators only see each other's events. Solved using session:${"{sessionId}"} Socket.IO rooms.
  • JWT + Socket.IO auth — Socket doesn't support cookies; JWT passed in handshake auth field and verified in middleware.
  • Single-device login — Tracked a loginSessionId in the database and validated it on every socket connection.
  • Environment file privacy — Ensuring .env file syncs functionally but values are masked for the partner — required tracking the envOwner socket ID.
QHow long did the development take?

The full platform was built over approximately 6–8 weeks of active development:

Phase Work Done
Week 1–2 Authentication, matchmaking engine, basic collaboration page
Week 3–4 IDE integration (Monaco + WebContainers), file tree, terminal
Week 5–6 Quick Chat, Friends system, DM, AI assistant

Performance & Security

QHow do you ensure system performance?
  • WebContainers — Code runs client-side, zero server CPU for code execution
  • Debounced sync — File changes batched every 1 second, not on every keystroke
  • Monaco model caching — File models cached; switching tabs is instant
  • React state minimization — Terminal output uses useRef to avoid re-renders on each character
  • Rate limiting — All API endpoints rate-limited via express-rate-limit
  • Indexed MongoDB queries — Key lookup fields (userId, participants, sessionId) are indexed
QWhat security mechanisms are implemented?
Mechanism Implementation
Authentication JWT (HS256) with expiry time
Single-device login loginSessionId in DB, validated on every socket connection
CORS Whitelisted origins only (Vercel URL + localhost)
Rate limiting express-rate-limit on all routes
Input validation express-validator for all POST endpoints
Content moderation Chat messages scanned for harmful content before relay
Password hashing bcryptjs with salt rounds
Env file privacy Values masked for partner; excluded from ZIP downloads
File locking Prevents two users from editing the same file simultaneously
Code sandboxing WebContainers run in WASM browser sandbox — no server-side execution
QHow do you prevent malicious code execution?
  • WebContainers are sandboxed — Code runs in a browser WASM sandbox, completely isolated from the server and other users' environments
  • Each user has their own WebContainer — A malicious script cannot reach another user's session
  • No server-side execution — The backend (Render) never executes user code
  • Content moderation — Chat messages filtered for harmful content before delivery
  • File extension policies — Dangerous file types can be blocked at the IDE level

Scalability Questions

QCan this system support thousands of users?

Currently, the single Render instance handles hundreds of simultaneous WebSocket connections comfortably. For thousands of users, the architecture would evolve as follows:

  • The Node.js + Socket.IO backend would be clustered using PM2 or Docker containers
  • MongoDB Atlas already scales horizontally with replica sets
  • WebContainers run on the client — code execution adds zero server load regardless of user count — this is PairOn's key scaling advantage
QHow would you scale the platform in the future?
Users → Cloudflare CDN
     → Load Balancer (AWS ALB)
     → Multiple Node.js instances (ECS/EC2)
     → Redis (Socket.IO Adapter + caching)
     → MongoDB Atlas (M10+ cluster)
     → S3 (for future file/project storage)
     → Groq/OpenAI API (AI layer)

Key step: replacing in-memory Socket.IO rooms with a Redis Adapter allows multiple backend instances to share socket state and event routing.

Limitations

QWhat are the limitations of your project?
  • Browser support — WebContainers only work fully on Chromium-based browsers (Chrome, Edge). Firefox and Safari have limited compatibility.
  • Node.js ecosystem only — Languages like Python, Java, Go, and C++ cannot run in the browser WebContainer (only JavaScript/Node.js ecosystem).
  • Isolated WebContainers — Each user has their own WebContainer; npm install in one user's environment doesn't automatically install in the partner's. They must each run it independently.
  • Performance on low-end devices — Running Monaco + WebContainers + WebSockets simultaneously requires ~1–2GB RAM. May be slow on budget devices.
  • Cold start delay — WebContainers take 20–30 seconds to initialize on first load.
  • No video/voice calls — Collaboration is text and code only; no built-in video or audio calling.
  • localStorage persistence — Project files stored in browser localStorage, not in a cloud file system, so they don't transfer between devices.
QWhat problems still remain unsolved?
  • True shared filesystem — Sharing one filesystem between two browser WebContainer instances is architecturally impossible with the current WASM design. This is a fundamental limitation of the WebContainers API itself, not just our implementation (same constraint StackBlitz faces).
  • Code conflict resolution — If two users edit the same file line simultaneously, last write wins. No CRDT/OT-based merging (like Google Docs uses) is implemented yet.
  • Mobile support — Monaco Editor and WebContainers don't function well on mobile browsers.

Final Inspection Questions

QWhat did you learn from building this project?
  • Real-time systems are hard — Designing for race conditions, reconnections, and stale state in multiplayer apps requires very careful architecture.
  • Browser capabilities are expanding — WebContainers proved that running a full Node.js server inside a browser tab is now possible, opening new product categories.
  • Security at every layer — Even small features (like syncing .env files) have serious security implications that must be thought through carefully.
  • Optimistic UI matters — Immediate feedback (showing a message before server confirms) dramatically improves perceived performance and user satisfaction.
  • Monaco is complex — Properly managing multi-file models, language servers, type stubs, and diagnostic suppression took significant effort.
  • Design before code — Several features needed to be rearchitected because the initial design didn't account for edge cases in real-time collaboration.
QWhat would you do differently if you rebuilt the project?
  • Use CRDT (Yjs or Automerge) from day one for code sync instead of last-write-wins
  • Separate the Socket.IO server from the REST API from the start — cleaner architecture and independent scaling
  • Design the file system as a tree from day one, not a flat key-value map
  • Use a shared cloud filesystem (S3/R2) instead of localStorage for project persistence
  • Add end-to-end tests from the beginning using Playwright or Cypress
  • Use TypeScript strict mode everywhere from day one to catch bugs earlier in development
